Apex Enterprise Design Patterns

These patterns form the foundation of a scalable, maintainable, and testable codebase. They promote a separation of concerns, ensuring each part of your code has a distinct and predictable responsibility. A typical invocation flows from the Trigger -> Handler -> Service -> Domain -> Selector.

  • Service Layer: Encapsulates business logic. Service methods should be atomic and represent a single business operation (e.g., `escalateCases`). They coordinate calls to other layers but do not contain SOQL or DML themselves.
  • Domain Layer: Encapsulates all logic for a single sObject, such as complex validation, default values, and record-specific calculations. It's the home for logic that belongs to the object itself, regardless of where it's called from.
  • Selector Layer: The only place where SOQL queries should live. This pattern centralizes data access, making it easy to optimize, secure, and reuse queries. Selectors return lists or maps of sObjects.
  • Unit of Work: Manages DML operations to ensure transactional integrity and respect governor limits. Instead of performing DML directly, you "register" records with the Unit of Work, which then performs a single, bulk DML operation at the end of the transaction.

Example: How Patterns Work Together

// 1. Service Layer orchestrates the operation
public class OpportunityService {
    public static void closeWonOpportunities(List<Id> oppIds) {
        fflib_SObjectUnitOfWork uow = new fflib_SObjectUnitOfWork(/*schema*/);
        // 2. Selector Layer fetches the data
        List<Opportunity> opps = OpportunitiesSelector.selectByIds(new Set<Id>(oppIds));

        // 3. Domain Layer contains the object-specific logic
        Opportunities oppsDomain = new Opportunities(opps);
        oppsDomain.handleCloseWon(uow); // Domain method registers DML with UoW

        // 4. Unit of Work commits all changes at once
        uow.commitWork();
    }
}

// 3. Domain Class
public class Opportunities {
    private List<Opportunity> records;
    public Opportunities(List<Opportunity> records) { this.records = records; }

    public void handleCloseWon(fflib_SObjectUnitOfWork uow) {
        List<Task> followUpTasks = new List<Task>();
        for(Opportunity opp : this.records) {
            opp.StageName = 'Closed Won';
            opp.CloseDate = Date.today();
            // Register the update with the Unit of Work
            uow.registerDirty(opp); 
            // Create related records
            uow.registerNew(new Task(
                Subject = 'Follow up on Won Opp: ' + opp.Name, 
                WhatId = opp.Id
            ));
        }
    }
}

Apex Trigger Frameworks

A robust trigger framework is essential for controlling execution order and managing complexity. The following patterns are standard in enterprise development.

Handler/Dispatcher Pattern with ITriggerHandler

This is the most common and recommended pattern. A single trigger per object calls a dispatcher class. The dispatcher then invokes a specific handler class for that object, which must implement the `ITriggerHandler` interface. This enforces a consistent structure across all handlers.

// ITriggerHandler Interface
public interface ITriggerHandler {
    void beforeInsert(List<SObject> newRecords);
    void beforeUpdate(Map<Id, SObject> newMap, Map<Id, SObject> oldMap);
    void afterInsert(Map<Id, SObject> newMap);
    // ... other context methods
}

// AccountTriggerHandler implements the interface
public class AccountTriggerHandler implements ITriggerHandler {
    public void beforeInsert(List<SObject> newRecords){
        // Logic here
    }
    // ... other required methods
}

// Trigger is still one line
trigger AccountTrigger on Account (before insert) { TriggerDispatcher.dispatch(); }

Trigger Factory Pattern

An evolution of the handler pattern, the Factory pattern uses an interface (`ITriggerHandler`) and a factory class to create instances of handlers. This is highly scalable and excellent for dependency injection and testing.

public class TriggerFactory {
    public static ITriggerHandler createHandler(SObjectType t) {
        if (t == Account.SObjectType) {
            return new AccountTriggerHandler();
        } // ... etc
        return null;
    }
}

// Trigger calls the factory
ITriggerHandler handler = TriggerFactory.createHandler(Trigger.SObjectType);
if(handler != null) { /* call appropriate method */ }

Asynchronous Apex

Asynchronous Apex is used to run processes in the background for long-running operations, callouts, and to avoid hitting governor limits.

TypeWhen to UseAdvantages
@futureSimple, fire-and-forget callouts or operations in a separate transaction.Very simple to implement for basic async needs.
QueueableChaining jobs, complex operations, and when you need more control than `@future`.Can chain one job to another. Can use non-primitive types. Gets a job ID for monitoring.
BatchableProcessing thousands or millions of records.Breaks large data sets into manageable chunks, each with its own governor limits.
SchedulableRunning a job at a specific time or on a recurring schedule.Reliable, declarative scheduling through the UI or can be scheduled programmatically.

Example: Queueable Apex for Chaining

public class ProcessAccountsQueueable implements Queueable {
    private List<Account> accounts;
    public ProcessAccountsQueueable(List<Account> records) { this.accounts = records; }

    public void execute(QueueableContext context) {
        // Perform processing on accounts
        // ...
        
        // Chain to the next job if needed
        if (!System.isBatch()) { // Avoid chaining from a batch context if not desired
            System.enqueueJob(new NotifyContactsQueueable(accounts));
        }
    }
}

Apex Security Best Practices

Writing secure Apex is non-negotiable. The two primary pillars are respecting user permissions (Sharing, CRUD, FLS) and preventing injection vulnerabilities.

Reusable FLSUtilities Class

Never perform DML or a query without first checking the user's permissions. This utility class makes this easy and reusable.

public with sharing class FLSUtilities {

    public static Boolean isCreateableRecord(String sObjectName, List<String> fieldsToCheck) {
        SObjectType schemaType = Schema.getGlobalDescribe().get(sObjectName);
        if(!schemaType.getDescribe().isCreateable()) return false;
        
        Map<String, Schema.SObjectField> fieldMap = schemaType.getDescribe().fields.getMap();
        for (String fieldName : fieldsToCheck) {
            if (!fieldMap.get(fieldName.toLowerCase()).getDescribe().isCreateable()) {
                return false;
            }
        }
        return true;
    }

    public static Boolean isUpdateableRecord(String sObjectName, List<String> fieldsToCheck) {
        SObjectType schemaType = Schema.getGlobalDescribe().get(sObjectName);
        if(!schemaType.getDescribe().isUpdateable()) return false;

        Map<String, Schema.SObjectField> fieldMap = schemaType.getDescribe().fields.getMap();
        for (String fieldName : fieldsToCheck) {
            if (!fieldMap.get(fieldName.toLowerCase()).getDescribe().isUpdateable()) {
                return false;
            }
        }
        return true;
    }
    
    public static Boolean isDeletableRecord(String sObjectName) {
        SObjectType schemaType = Schema.getGlobalDescribe().get(sObjectName);
        return schemaType.getDescribe().isAccessible() && schemaType.getDescribe().isDeletable();
    }
}

SOQL with SECURITY_ENFORCED

This is the modern, preferred way to enforce Field-Level Security directly within a SOQL query. It automatically checks if the running user has access to the fields being queried and will throw an exception if they do not. This simplifies security checks significantly compared to manual describe checks for every field.

// This query will FAIL if the user doesn't have FLS to see AnnualRevenue.
try {
    List<Account> accs = [
        SELECT Id, Name, AnnualRevenue 
        FROM Account 
        WITH SECURITY_ENFORCED
    ];
} catch (System.QueryException e) {
    // Handle the security exception gracefully
    System.debug('User does not have permission to view all fields.');
}

Bypass Validation Rules Utility

This utility provides a framework for programmatically bypassing validation rules, typically for data migrations or system integrations. It relies on a custom checkbox field on the object (`ByPass_Validation_Rules__c`) and another date/time field (`Last_Bypass_Date__c`) to trigger the bypass.

public with sharing class ByPassValidationRulesUtil {
    public static void updateByPassValidationsFields(SObject oldObject, SObject newObject) {
        Datetime lastByPassDate = (Datetime) newObject.get('Last_Bypass_date__c');
        Datetime prevByPassDate = oldObject != null ? (Datetime) oldObject.get('Last_Bypass_date__c') : null;
        Boolean byPassValidationRules = lastByPassDate != null && lastByPassDate != prevByPassDate;
        newObject.put('ByPass_Validation_Rules__c', byPassValidationRules);
    }
}

Advanced Apex Concepts

Managing Trigger Recursion

Trigger recursion occurs when a DML operation inside a trigger causes the same trigger to fire again. A simple way to prevent this is a static boolean flag. The flag is set to `true` the first time the trigger runs, and subsequent executions in the same transaction are skipped. It's crucial to only set this for the specific trigger to avoid unintentionally disabling other automations.

public class AccountTriggerHandler {
    private static boolean hasRun = false;
    public void afterUpdate(Map<Id, SObject> newMap, Map<Id, SObject> oldMap) {
        if(hasRun){ return; }
        hasRun = true;
        // ... Logic that might cause recursion
    }
}

Managing Concurrency and Record Locking

When two separate transactions try to update the same record simultaneously, a race condition can occur. To prevent this, you can lock the record for update. The `FOR UPDATE` clause in a SOQL query places a lock on the queried records, preventing any other transaction from updating them until the current transaction is complete. This is critical for processes that update parent records based on child record changes.

// In a trigger on a child object (e.g. Opportunity)
Set<Id> accountIds = new Set<Id>();
for(Opportunity opp : (List<Opportunity>)Trigger.new) {
    accountIds.add(opp.AccountId);
}
// Lock the parent accounts before updating them to prevent conflicts.
List<Account> lockedAccounts = [SELECT Id FROM Account WHERE Id IN :accountIds FOR UPDATE];
// Now it's safe to perform roll-up calculations and update the accounts.

Writing Apex for Large Data Volumes (LDV)

When working with objects that have millions of records, performance is paramount. Every query must be highly selective. Use the Query Plan Tool to analyze query performance. Never query fields you don't need. For processing large sets of data, always use Batch Apex or `SOQL For Loops` to avoid hitting heap size limits.

Debugging Best Practices

Effective debugging goes beyond `System.debug`. Use **Checkpoints** in the Developer Console to inspect the heap at a specific point in your code. Set up **Debug Levels** to control the verbosity of logs. Use the **`Limits` class** to output governor limit consumption at various points in your transaction to pinpoint bottlenecks.

LWC Lifecycle Hooks & DOM Concepts

The LWC framework provides methods, called lifecycle hooks, that allow you to tie into the lifecycle of a component. Understanding these is critical for writing correct and performant code. The core concept is the Shadow DOM, which encapsulates the component's structure, style, and behavior, keeping it separate from other components.

  • constructor(): Called when the component is first created. Use it for basic initialization. You cannot access the component's public properties or the DOM here.
  • connectedCallback(): Called when the component is inserted into the DOM. This is the main hook to perform initialization tasks, fetch data, or set up event listeners.
  • renderedCallback(): Called after the component's template has been rendered. Use this hook to interact with the UI (the rendered DOM). Be careful, as this can get called many times; use a boolean flag to ensure logic runs only once if needed.
  • disconnectedCallback(): Called when the component is removed from the DOM. Use this to clean up, e.g., by removing event listeners.
  • errorCallback(error, stack): A special hook that acts as a try-catch for all your component's child components. It captures errors that occur during any lifecycle hook of its descendants.

Advanced Recipe: Datatable with Chart.js

This recipe shows how to fetch data via Apex and visualize it in a chart alongside a standard datatable. First, upload the Chart.js library as a static resource named `chartjs_v3`. This component will display Opportunities grouped by stage and show a doughnut chart of the stage distribution.

<!-- opportunityChartTable.html -->
<template>
    <lightning-card title="Opportunity Stage Analysis" icon-name="standard:dashboard">
        <div class="slds-grid slds-wrap slds-p-around_medium">
            <div class="slds-col slds-size_1-of-1 slds-medium-size_2-of-3 slds-p-right_medium">
                <lightning-datatable key-field="StageName" data={chartData} columns={columns}></lightning-datatable>
            </div>
            <div class="slds-col slds-size_1-of-1 slds-medium-size_1-of-3">
                <div class="chart-container" style="position: relative; height:250px; width:100%">
                    <canvas class="doughnut-chart"></canvas>
                </div>
            </div>
        </div>
    </lightning-card>
</template>

// opportunityChartTable.js
import { LightningElement, wire } from 'lwc';
import { loadScript } from 'lightning/platformResourceLoader';
import chartjs from '@salesforce/resourceUrl/chartjs_v3';
import getOpptyStageSummary from '@salesforce/apex/OpportunityController.getOpptyStageSummary';

const columns = [
    { label: 'Stage', fieldName: 'StageName' },
    { label: 'Total Amount', fieldName: 'TotalAmount', type: 'currency' },
    { label: 'Record Count', fieldName: 'RecordCount', type: 'number' },
];

export default class OpportunityChartTable extends LightningElement {
    columns = columns;
    chartData;
    chart;
    chartjsInitialized = false;

    @wire(getOpptyStageSummary)
    wiredData({ error, data }) {
        if (data) {
            this.chartData = data;
            if (this.chartjsInitialized) {
                this.updateChart();
            }
        } else if (error) {
            console.error(error);
        }
    }

    renderedCallback() {
        if (this.chartjsInitialized) {
            return;
        }
        this.chartjsInitialized = true;

        loadScript(this, chartjs)
            .then(() => {
                this.initializeChart();
            })
            .catch(error => {
                console.error('Error loading Chart.js', error);
            });
    }

    initializeChart() {
        const ctx = this.template.querySelector('canvas.doughnut-chart').getContext('2d');
        this.chart = new window.Chart(ctx, {
            type: 'doughnut',
            data: {
                labels: [],
                datasets: [{
                    data: [],
                    backgroundColor: ['#003972', '#005fb2', '#0099de', '#44a7e0', '#d8dde6'],
                }]
            },
            options: {
                responsive: true,
                maintainAspectRatio: false,
                legend: { position: 'right' }
            }
        });
        if (this.chartData) {
            this.updateChart();
        }
    }

    updateChart(){
        if (!this.chart || !this.chartData) return;
        this.chart.data.labels = this.chartData.map(row => row.StageName);
        this.chart.data.datasets[0].data = this.chartData.map(row => row.TotalAmount);
        this.chart.update();
    }
}

Matching Rules define how to identify duplicate records by comparing field values, using either exact matching or fuzzy logic (e.g., for names and addresses). Duplicate Rules use these matching rules to determine what happens when a user tries to create a duplicate record. Code Guidelines: Apex

Introduction

This document serves as the complete definition of VRP's coding standards for source code in the APEX™ Programming Language. Like other programming style guides, the issues covered span not only aesthetic issues of formatting but other types of conventions or coding standards as well. However, this document focuses primarily on the hard-and-fast rules that we follow universally and avoids giving advice that isn't clearly enforceable (whether by human or tool). Guideline items marked with (!) are considered to be CRITICAL and obligatory to follow.

Formatting

  • APEX class doc

    You can find all the instructions for creating apex-doc in Apex Doc github repository. For sure, almost no one uses doc generation to html nowadays. But leaving the comments in a documented format could be parsed by your IDE and shown in a more readable way. The text below is an example. Some notes about documenting

    • @description tag can be omitted
    • There is a @see tag that can be parsed by IntelliJ's Illuminated Cloud that allows you to refer to another class (but not class methods) for details.
    • For VS Code users: you can take a look at Apex Doc's inheritor, Apex Dox for vs code. It contains some more features and even supports inline {@link} tag.

    Further sample docs are created using the original Apex Doc notation to maintain overall usability.

    /**
 * @description Guitar Model Object. Various attributes of guitars, and related behavior.
 * Note that Decimal is used to model the price - not Double or Float.
 * @author Les Paul
 * @version 2.0
 */
  • New lines, indent conventions, and spacing (!)
    • Avoid concatenating two or more lines of code in one (more than one end of code-line symbol ';' per line).
    • Wrap all operators =, >, <, -, +, etc. with exactly ONE space.
    • Never use more than one space in the code.
    • You can add one line after class/trigger initialization and one before the last curly bracket, just to make the class more readable.
    • Every code block enclosed in curly brackets { } has to be marked with 4 spaces indent from parent level code block.
    // Correct
public class TestRunAs {
    public static testMethod void testRunAs() {
        ...
    }
}
// Incorrect
public class TestRunAs {public static testMethod void testRunAs() {...}}
    • Do use a single space after a comma between method arguments.
    • Do not add a single space before the argument and after the parenthesis. The same is for the last argument - do not add space between the last argument and parenthesis.
    • Do not use spaces between a method name and parenthesis in method execution.
    // Correct
Foo.bar(myChar, 0, 1);
// Incorrect
Foo.bar(myChar,0,1);
Foo.bar( my Char, 0, 1 );
Foo.bar (myChar, 0, 1);
  • Use the multiline full version of getter and setter methods. If both getter and setter are auto-generated, use the same line for them. Use single space between the getter/setter definition and braces. 
    // Correct
public String name {
    get {
        Integer a = 100;
        a++;
        return 100-a;
    }
    set;

Naming conventions (!)

  • Rules by identifier type
    • Class / Interface - Names should be nouns, in UpperCamelCase. - Names should be simple and descriptive. - Use whole words - avoid acronyms and abbreviations (unless the abbreviation is much more widely used than the long form, such as URL or HTML).
    • Method - Names should be verbs, in lowerCamelCase. Examples: run();, getBackground();
    • Variable - Names should be short yet meaningful. - The choice of a variable name should be mnemonic - designed to indicate to the casual observer the intent of its use. - One-character variable names should be avoided except for temporary and looping variables.
    • Constants - Should be UPPER_CASE with words separated by an underscore character.
    • Batch Apex Classes - Should have Batch at the end. Example: AccountRefundBatch.
    • Controllers - Can have Controller or Ctr postfix. Example: SolarEnergyController, SolarEnergyCtr. - Can have Extension or Ext postfix. Example: SolarEnergyExtension, SolarEnergyExt.
    • Triggers / Trigger Handlers - Should have Trigger postfix. Example: AccountTrigger. - Handlers should have TriggerHandler postfix. Example: AccountTriggerHandler.
    • Test Classes - Should have Test postfix. Example: CreditAccountTest.

Code structure

  • Sizes
    • One statement per line.
    • Preferable row length: 120 characters (could vary 80 ~ 140 characters).
    • Identify duplicate code fragments and wrap them into methods.
    • Preferable method size: up to 50 rows. Usually rule 'shorter method - better method' works fine.
    • Preferable parameters amount in a method is up to 2. Use OOP and class variables. With this approach, you usually don't need methods with parameters!
    • Avoid any references to strings or numbers in the methods. Use constants for that. Exceptions: Apex Class name, VF Page name, VF Component name, and other API / Developer names.
    • All class fields should be placed at the top of the class definition.
  • Variables
    • Try to keep variables scoped to the lowest level possible. Global variables should be avoided.

Refactoring (!)

  • Refactoring is a mandatory part of development that should be performed after logic is ready (!).
  • Remove unused code fragments.
  • Before code release get rid of all "junk" code and temporary comments.
  • Make sure variables' names are short clear and meaningful for a casual reader.
  • Make sure the code style is fine.
  • Make sure your algorithms are clear and not overwhelmed with logic.
  • Reduce dependencies in the code. It's fine to create several small classes with clear logic to reduce dependencies.
  • Avoid multi-purpose classes and methods.
  • Avoid more than 3 level nesting in loops and conditions.
  • Use the KISS (keep it simple, stupid) approach

Exception handling

  • Catching - All the exceptions in Apex class that can be thrown should be covered to prevent the abnormal direction of execution flow.
  • Throwing - Standard exceptions should be used wherever it is possible. - Exceptions are not used to control application flow 
    // Correct
if (String.isNotBlank(someString)) {
    someString = someString.toLowerCase();
} else {
    someString = EMPTY_STRING;
}

// Incorrect

    someString = 'empty';
}

  • Usage of bulk data processing (!) 
    // Correct

  • No SOQL inside loops (!)
  • Usage of efficient Queries (relationship queries) and minimizing the number of loops
  • Minimising number of triggers (streamlining triggers for one object).
  • Query large data sets in for loops

    Let the Force.com platform chunk your large query results into batches of 200 records by using this syntax where the SOQL query is in the for loop definition, and then handle the individual datasets in the for loop logic.

    // Use this format for efficiency if you are executing DML statements

    //  Your code here
}

  • Break loops if the further iteration is not necessary
  • Logic that always gets the same results does not occur within the body of loops
  • Recursion should always have a "hand brake"

Governor Limits

  • Use of the Limits methods to avoid hitting Governor Limits.
  • Soql - Total queries issued: 100 (200 for async apex) - Total records retrieved: 50000 - Non-selective queries: The selectivity threshold is 10% of the records for the first million records and less than 5% of the records after the first million records, up to a maximum of 333,000 records. - Query length: 10000 symbols - Order by fields: 32 Limits on querying for ContentDocumentLink, NewsFeed, KnowledgeArticleVersion, UserProfileFeed, Vote
  • Sosl - Total number of SOSL queries issued: 20
  • Dml - Total number of DML: 150 - Total stack depth for Apex invocation after DML statement: 16
  • Callouts - Total number of Callouts (HTTP requests to external service): 100 - Maximum timeout for Callout: 120 seconds - Maximum size of callout request: 3Mb (use gZip compression if possible)
  • Future calls - Total number of future calls per Apex invocation: 50 - Do not call @future methods in loops . Invoke @future for batch of records instead - Should be bulkified - working with set / list of data instead of single record
  • Email - Total number of sendEmail method calls: 10
  • Heap size - Total heap size: 6Mb - Total heap size for async APEX: 12Mb - Email services heap size: 36Mb
  • CPU - Maximum CPU time on the Salesforce servers: 10 seconds (60 seconds for async apex) - Execution time spent in the database for DML, SOQL, SOSL, or waiting Apex callouts isn't counted as CPU time.

Security

  • Injections - Check, validate, and escape (String.escapeSingleQuotes()) all the data in dynamic queries that are coming from client side.
  • Storing secrets
    • Private Class Solution. - Class with the secret should be defined as private - Additional constructor should be defined with some parameters - they can be fake or real (calculations can be made based on this parameter or just check for "right" value) - No getter and setter for the secret key. - Method that will return boolean conditional check variable (input parameter pass validation or not) - so end users will be not able to store parameters (like hashes/keys) for further investigation.
    • Private Custom Setting Solution - Custom Setting should be pre-filled during package installation (Post Installation script) - All calculations should be done in private classes - Method that will return boolean conditional check variable (input parameter pass validation or not) - so end users will be not able to store parameters (like hashes/keys) for further investigation.
    • Private Data - All the private data on the Salesforce DB side (e.g CreditCard Data, Bank Account Data..) should be stored in encrypted fields
  • Access control - Use with sharing model for controllers that retrieve user-related data from the database
  • Cross-site scripting - All standard Visualforce components, which start with<apex>, have anti-XSS filters in place. For example, the following code is normally vulnerable to an XSS attack because it takes user-supplied input and outputs it directly back to the user, but the <apex:outputText> tag is XSS-safe. All characters that appear to be HTML tags are converted to their literal form. For example, the < character is converted to < so that a literal < displays on the user's screen. Use the escape parameter (escape="false") only in extreme cases because with this parameter your code becomes more vulnerable.
  • With / without sharing - Use the with sharing keywords when declaring a class to enforce the sharing rules that apply to the current user. Use the without sharing keywords when declaring a class to ensure that the sharing rules for the current user are not enforced. Using the with sharing keywords is preferable.

Unit tests

  • Unit tests should check if the desired behavior is a case.
  • Unit test should NOT bet just a list of controller methods for execution.
  • Unit test should be created for a real-life scenario with appropriate logic - asserts should be used.
  • Test data should be generated in separate methods.
  • `isTest(SeeAllData = true)` attribute is prohibited. (!)
  • Data generation logic should be reusable in all of the test-methods related to this class/group of classes - so you need to create methods that will generate desired data, instead of hardcoding data creation in each test method.
  • Separate unit test classes are implemented instead of member test-methods in tested class
  • Stress-tests - Test classes should have stress-test methods when a huge amount of data will be generated - this scenario will help find the breaches in the logic for "real-life" scenarios.

Code Guidelines: LWC (HTML / CSS)

Indentation

  • Indent by 2 or 4 spaces depending on project.
  • Don’t use tabs or mix tabs and spaces for indentation.
<ul>
  <li>Fantastic</li>
  <li>Great</li>
</ul>
.example {
  color: blue;
}

Capitalization

  • Use only lowercase.
  • All code has to be lowercase: This applies to HTML element names, attributes, attribute values (unless text/CDATA), CSS selectors, properties, and property values (with the exception of strings).
<!-- Recommended -->
<img src="google.png" alt="Google">
/* Not recommended */
color: #E5E5E5;

/* Recommended */
color: #e5e5e5;

Trailing Whitespace

  • Remove trailing white spaces.
  • Trailing white spaces are unnecessary and can complicate diffs.
<!-- Recommended -->
<p>Yes please.</p>

<!-- Not recommended -->
<p>What?_</p>

HTML Validity

  • Use valid HTML where possible.
  • Use tools such as the W3C HTML validator to test HTML validity.

Semantics

  • Use HTML according to its purpose.
  • Using HTML according to its purpose. For example, use p elements for paragraphs, a elements for anchors, etc.
<!-- Recommended -->
<a href="recommendations/">All recommendations</a>

<!-- Not recommended -->
<div onclick="goToRecommendations();">All recommendations</div>

Entity References

  • Usually there is no need to use entity references like &mdash; or &#x263a;, assuming the same encoding (UTF-8) is used for files and editors as well as among teams. The only exceptions apply to characters with special meaning in HTML (like < and &) as well as control or "invisible" characters (like no-break spaces).
<!-- Recommended -->
The currency symbol for the Euro is “€”.

<!-- Not recommended -->
The currency symbol for the Euro is &ldquo;&eur;&rdquo;.

Line-Wrapping

  • Preferable row length: 120 characters (could vary 80 ~ 140 characters).
  • Use line wrapping if row is too long.
<!-- Possible way -->
<md-progress-circular md-mode="indeterminate" class="md-accent"
    ng-show="ctrl.loading" md-diameter="35"></md-progress-circular>

<!-- Possible way -->
<md-progress-circular
    md-mode="indeterminate"
    class="md-accent"
    ng-show="ctrl.loading"
    md-diameter="35"></md-progress-circular>

HTML Quotation Marks

  • Use double ("") rather than single quotation marks ('') around attribute values.
<!-- Recommended -->
<a class="maia-button maia-button-secondary">Login</a>

<!-- Not recommended -->
<a class='maia-button maia-button-secondary'>Login</a>
